3.1  CPU  Model  
●  Architecture:  32‑bit  MIPS‑like  RISC  ○  32  general‑purpose  registers  (R0  hard‑wired  to  0).  ○  No  hardware  ﬂoating‑point  unit  (no  FPU).  ○  Register‑based  calling  convention  (similar  to  standard  MIPS).  
 ●  Pipeline:  ○  Classic  5‑stage  pipeline:  IF,  ID,  EX,  MEM,  WB.  ○  Hazards:  ■  Load‑use  hazard  incurs  1‑cycle  stall  if  not  avoided.  ■  Taken  branches  incur  1‑cycle  penalty  (ﬂush).  ■  Multiplies/divides  are  multi‑cycle  (conﬁgurable  cost).  ●  Instruction  timing  (example  constants):  ○  ALU  ops  (ADD/SUB/AND/OR/XOR,  shifts):  1  cycle.  ○  Branch  (BEQ/BNE)  +  pipeline  penalty  if  taken:  1  +  1  cycles.  ○  Load:  1  cycle  +  potential  memory  penalty.  ○  Store:  1  cycle  +  potential  memory  penalty.  ○  Multiply:  3  cycles.  
 
3.2  Memory  Model  

===== PAGE 6 =====
●  Main  memory:  ○  Total  RAM:  128  KB  (Based  on  requests,  increasing  this  may  be  considered).  ○  Code  +  data  must  ﬁt  within  this  limit.  ●  Memory  regions  (no  MMU):  ○  .text:  program  code.  ○  .data/.bss:  global  data,  static  buffers.  ○  Stack:  downward‑growing  region  per  task.  ●  Optional  cache  (simpliﬁed):  ○  4  KB  “fast  memory”  region  with  no  extra  access  penalty.  ○  All  other  accesses  cost  an  extra  +3  cycles  (simulating  off‑chip  RAM).  ○  Teams  can  choose  to  place  critical  data  structures.

●  Core  logic  in  C,  with  optional  hand‑tuned  assembly  for  critical  kernels.

4.1  Task  Model

time

 ○  Classify  risk:  SAFE  /  WATCH  /  HIGH‑RISK.  3.  T3:  Maneuver  Planning  Task  ○  Aperiodic,  triggered  when  T2  ﬁnds  HIGH‑RISK  objects.  ○  Evaluates  a  small  discrete  set  of  candidate  maneuvers:  ■  No  maneuver.  ■  Along‑track  ±Δ v  (e.g.,  +3  cm/s,  −3  cm/s).  ■  Radius  ±Δ r  (eg.,  +1m,  -1m)  in  orbital  radius.  ○  Uses  quick  propagation  to  estimate  which  candidate  increases  separation

■  Resource  usage  (CPU,  energy  estimates).  ○  Marks  radio  as  active  during  transmission  (energy  penalty).  5.  ●  Comments  explaining  ISA‑level  decisions  (e.g.,  branch  patterns,  pipeline  
concerns).
